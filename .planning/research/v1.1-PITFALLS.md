# Domain Pitfalls

**Domain:** Adding Open Notebook + DB-GPT to existing AGMind macOS Docker Compose installer
**Researched:** 2026-03-24
**Confidence:** MEDIUM (Open Notebook well-documented; DB-GPT macOS specifics require validation)

## Critical Pitfalls

Mistakes that cause rewrites, data loss, or broken installations.

---

### Pitfall 1: DB-GPT Docker Image -- No Confirmed ARM64 Build

**What goes wrong:** DB-GPT official Docker images (`eosphorosai/dbgpt-openai`, `eosphorosai/dbgpt`) are built primarily for amd64 (x86_64). On Apple Silicon, Docker must emulate via Rosetta/QEMU, causing 3-5x slowdown, random crashes, and increased memory usage. Some images may fail to start entirely.

**Why it happens:** DB-GPT targets NVIDIA GPU servers (Linux amd64). The "openai" (CPU proxy) variant may lack multi-arch builds.

**Consequences:** Container starts but crashes under load; or works in testing but fails with real workloads. User experience degrades silently -- performance looks like an LLM problem, not an architecture problem.

**Prevention:**
- Before adding to compose: `docker manifest inspect eosphorosai/dbgpt-openai:latest | grep architecture` to verify arm64 support
- If no arm64: document the Rosetta emulation penalty in wizard warning
- Add `platform: linux/amd64` explicitly in compose to force emulation rather than cryptic startup failures
- Consider offering source-code install via pip as alternative for Apple Silicon
- Phase: **Preflight** (detect architecture, warn user)

**Detection:** `docker inspect <container> | grep Architecture` shows "amd64" on an arm64 host = emulation active. High CPU usage for simple operations.

**Confidence:** LOW -- Docker Hub page could not be scraped for architecture info; needs manual verification before implementation.

### Pitfall 2: DB-GPT Requires MySQL -- AGMind PostgreSQL Cannot Be Shared

**What goes wrong:** DB-GPT's Docker Compose ships with MySQL as its database. The AGMind stack uses PostgreSQL. Attempting to point DB-GPT at the existing PostgreSQL instance will fail -- DB-GPT's ORM and migrations are MySQL-specific.

**Why it happens:** DB-GPT supports SQLite (default, embedded) or MySQL. PostgreSQL is not supported. Developers assume they can consolidate databases.

**Consequences:** Either a second database service (MySQL) bloats the stack, or the developer wastes time trying to make PostgreSQL work.

**Prevention:**
- Use DB-GPT's **built-in SQLite** mode (default) -- no external database needed. This eliminates the MySQL container entirely.
- If MySQL is required for advanced features: add `dbgpt-mysql` as a separate profile-gated service, NOT the existing `db_postgres`
- Never share the PostgreSQL container with DB-GPT
- Phase: **Config Generation** (choose SQLite mode, generate correct env vars)

**Detection:** DB-GPT logs show "Access denied" or "Unknown database engine" errors.

**Confidence:** HIGH -- DB-GPT docs explicitly state SQLite default, MySQL optional, no PostgreSQL.

### Pitfall 3: Port Collision Map -- SurrealDB, Open Notebook, DB-GPT vs Existing Stack

**What goes wrong:** Port conflicts between new and existing services. Current AGMind stack port map:

| Port | Service | Status |
|------|---------|--------|
| 80 | nginx | **Used** |
| 3000 | dify-web | Internal (no host bind) |
| 5001 | dify-api | Internal (no host bind) |
| 5002 | plugin_daemon | **Used (host bind)** |
| 5670 | DB-GPT webserver | **NEW** |
| 6333 | qdrant | Internal |
| 8000 | SurrealDB | **NEW** |
| 8080 | weaviate | Internal |
| 8502 | Open Notebook web UI | **NEW** |
| 5055 | Open Notebook REST API | **NEW** |
| 9000 | portainer | Conditional |
| 11434 | ollama | Host native |

**Why it happens:** Each tool's default docker-compose exposes ports on the host. When merged into a shared compose file, ports conflict or unnecessarily expose internal-only ports to the host.

**Consequences:** `docker compose up` fails with "port already in use" or services bind to wrong ports. Worse: if both services start on the same port, one silently fails.

**Prevention:**
- Do NOT expose SurrealDB (8000), Open Notebook API (5055), or DB-GPT (5670) to the host. Route everything through nginx.
- Only nginx binds to host port 80. All new services are internal Docker network only.
- Add preflight port check for 8502 and 5670 if those ports need temporary host exposure during development.
- Phase: **Preflight** (port scan), **Config Generation** (nginx routes, no host port bindings for new services)

**Detection:** `lsof -iTCP:8000 -sTCP:LISTEN` before compose up.

**Confidence:** HIGH -- port map derived directly from existing docker-compose.yml and upstream documentation.

### Pitfall 4: Ollama Connection -- Each Tool Uses Different Config Mechanism

**What goes wrong:** Each tool uses a DIFFERENT mechanism to configure the Ollama API endpoint:

| Tool | Config Mechanism | Expected Value |
|------|-----------------|----------------|
| Dify | Env var `OLLAMA_API_BASE` | `http://host.docker.internal:11434` |
| Open WebUI | Env var `OLLAMA_BASE_URL` | `http://host.docker.internal:11434` |
| Open Notebook | **Web UI only** (Settings > Credentials) | `http://host.docker.internal:11434` |
| DB-GPT | **TOML config file** (`api_base` field) | `http://host.docker.internal:11434` |

**Why it happens:** No standard exists. Each project chose its own configuration method. Copy-pasting from one tool's docs to another silently fails.

**Consequences:** Tool starts but cannot reach Ollama. User sees "model not found" or "connection refused" errors. Hard to debug because the container is running.

**Prevention:**
- **Open Notebook:** Cannot pre-configure Ollama via env vars. Provider credentials are configured through the web UI after first launch. The installer must document this in post-install instructions: "Open Settings > AI Providers > Add Ollama with URL http://HOST_IP:11434". Note: inside Docker, use `host.docker.internal`, but when accessed via browser for UI config, the user enters the host's actual IP or `localhost`.
- **DB-GPT:** Uses a TOML config file with `api_base` under model definitions. Must generate and mount `dbgpt_config.toml` into the container.
- Both tools need `extra_hosts: ["host.docker.internal:host-gateway"]` in compose (critical for Colima).
- Phase: **Config Generation** (generate TOML for DB-GPT), **Post-Install** (document Open Notebook UI setup)

**Detection:** `docker exec <container> curl -sf http://host.docker.internal:11434/api/tags` should return JSON.

**Confidence:** MEDIUM -- Open Notebook UI-based config verified via docs; DB-GPT TOML structure from search results (docs.dbgpt.cn timed out on fetch).

### Pitfall 5: SurrealDB Data Durability -- Known Community Concerns

**What goes wrong:** SurrealDB has documented community concerns about data durability. Reports indicate SurrealDB may sacrifice fsync for benchmark performance, risking data loss on crash. Additionally, SurrealDB has had crash bugs (uncaught exceptions in the Net module).

**Why it happens:** SurrealDB is a relatively young database (v2.x). RocksDB backend configuration defaults may not be production-hardened.

**Consequences:** Notebook data (sources, annotations, generated content) lost after container restart or host crash.

**Prevention:**
- Pin SurrealDB to a specific tested version (e.g., `surrealdb/surrealdb:v2.2.1`), never use `:latest` or `:v2` floating tags
- Volume mount for data persistence: bind mount `./surreal_data:/mydata` (matches upstream compose pattern)
- Add the SurrealDB data directory to the AGMind backup system (`backup.sh`)
- Consider periodic `surreal export` as part of backup LaunchAgent
- Phase: **Config Generation** (pin version in versions.env), **Backup** (include SurrealDB data)

**Detection:** After container restart, check if data persists. If notebooks disappear, volume mount is broken.

**Confidence:** MEDIUM -- community concerns documented on Lobsters; no direct Open Notebook data loss reports found, but the risk is structural.

### Pitfall 6: Docker Compose Profiles -- depends_on with Profiled Services Breaks Compose

**What goes wrong:** If nginx `depends_on` lists `open-notebook` or `dbgpt`, but those services have a profile assigned, `docker compose up` fails with "service is required by service but is disabled" when the profile is not activated.

**Why it happens:** Docker Compose v2.19 introduced a breaking change. Before v2.20.2, `depends_on` could not reference profiled services. After v2.20.2, the `required: false` property was added to `depends_on`.

**Prevention:**
- Do NOT add `open-notebook` or `dbgpt` to nginx's `depends_on`. Nginx must start regardless of which optional tools are enabled.
- Use `required: false` syntax if Compose version supports it: `depends_on: { open-notebook: { condition: service_started, required: false } }`
- Verify Colima/Desktop ships Compose v2.20.2+. If not, avoid cross-profile depends_on entirely.
- Nginx config must handle missing upstreams gracefully (conditional template rendering)
- Phase: **Config Generation** (compose template), **Preflight** (check `docker compose version`)

**Detection:** `docker compose up` immediately errors with "service ... is disabled". Running `docker compose version` shows < 2.20.2.

**Confidence:** HIGH -- Docker Compose GitHub issue #10751 confirms this exact behavior.

### Pitfall 7: Nginx Upstream Fails When Optional Service Not Running

**What goes wrong:** If nginx.conf defines `upstream open-notebook { server open-notebook:8502; }` but the Open Notebook profile is not activated, nginx refuses to start because DNS resolution for `open-notebook` fails at startup.

**Why it happens:** Nginx resolves all upstream hostnames at startup time. Missing containers = missing DNS entries in Docker's internal DNS.

**Consequences:** Entire nginx (and therefore the whole web UI including Dify and Open WebUI) goes down because one optional service isn't running.

**Prevention:**
- Generate nginx.conf dynamically based on which profiles are activated. Only include upstream blocks for enabled services.
- Alternative: use `resolver 127.0.0.11 valid=30s;` (Docker internal DNS) with variables in proxy_pass to defer resolution: `set $open_notebook http://open-notebook:8502; proxy_pass $open_notebook;`
- The template rendering in `_render_nginx_conf()` must be extended to conditionally include blocks based on WIZARD_* variables.
- Phase: **Config Generation** (conditional nginx template rendering)

**Detection:** `docker compose logs nginx` shows "host not found in upstream" error.

**Confidence:** HIGH -- standard nginx behavior, well-documented, and would break the entire stack.

### Pitfall 8: Open Notebook Encryption Key -- Lost on Re-install Breaks All Credentials

**What goes wrong:** Open Notebook requires `OPEN_NOTEBOOK_ENCRYPTION_KEY` for encrypted credential storage (AI provider keys, Ollama connection settings configured through UI). If this key changes between re-installs, all stored credentials become unreadable.

**Why it happens:** AGMind generates secrets in `_load_or_generate_secrets()` and persists them in `credentials.txt`. If `OPEN_NOTEBOOK_ENCRYPTION_KEY` is not added to this system, re-installs generate a new key.

**Consequences:** After re-install, Open Notebook starts but all configured AI providers are broken. User must re-configure every provider through the UI (which is particularly painful because Ollama config is UI-only).

**Prevention:**
- Add `OPEN_NOTEBOOK_ENCRYPTION_KEY` to the credential whitelist in `_load_or_generate_secrets()` (the `case` statement on line ~47-51 of config.sh)
- Generate it alongside other secrets and persist in `credentials.txt`
- Inject into the Open Notebook container via env_file or environment block in compose
- Phase: **Config Generation** (extend secret management)

**Detection:** After re-install, Open Notebook shows "decryption error" or provider credentials appear empty.

**Confidence:** HIGH -- Open Notebook docs explicitly state encryption key requirement and its consequences.

### Pitfall 9: DB-GPT Model Configuration -- TOML File, Not Env Vars

**What goes wrong:** DB-GPT's model configuration uses a TOML file (`dbgpt_config.toml`), not environment variables. The installer cannot simply add `OLLAMA_BASE_URL=...` to .env and expect DB-GPT to discover Ollama.

**Why it happens:** DB-GPT is designed for flexible multi-model setups (proxy mode with SiliconFlow, local GPU models, Ollama). Configuration is file-based.

**Consequences:** DB-GPT starts but has no LLM configured. User sees an empty model list and thinks the tool is broken.

**Prevention:**
- Generate a `dbgpt_config.toml` file during Config Generation phase
- Mount it into the container: `volumes: ["./dbgpt_config.toml:/app/configs/dbgpt_config.toml"]`
- Pre-populate with Ollama model definitions using `api_base = "http://host.docker.internal:11434"`
- Reuse the same `LLM_MODEL` and `EMBED_MODEL` from wizard to populate the TOML
- Phase: **Config Generation** (new template: `templates/dbgpt_config.toml.template`)

**Detection:** DB-GPT web UI shows no available models after startup.

**Confidence:** MEDIUM -- DB-GPT TOML config structure verified via search results; exact mount path and field names need testing.

### Pitfall 10: COMPOSE_PROFILES String Assembly -- Fragile String Concatenation

**What goes wrong:** Current `_build_compose_profiles()` builds a comma-separated string: `postgresql,weaviate,etl-extended,monitoring`. Adding `open-notebook` and `dbgpt` profiles means extending this builder. Common bugs:
- Trailing comma: `postgresql,weaviate,open-notebook,`
- Double comma: `postgresql,,open-notebook`
- Profile name mismatch between compose file and code (e.g., `opennotebook` vs `open-notebook`)

**Why it happens:** String concatenation in Bash 3.2 without proper join logic. No associative arrays available.

**Prevention:**
- Profile names must exactly match between docker-compose.yml service `profiles:` definitions and `_build_compose_profiles()` output
- Add BATS tests: assert profile string format for every combination of optional services
- Use a consistent naming convention: `open-notebook` and `dbgpt` (hyphenated, lowercase)
- Phase: **Config Generation** (extend `_build_compose_profiles`), **Testing** (new BATS tests for all combinations)

**Detection:** `docker compose --profile open-notebook config` fails with "no such service".

**Confidence:** HIGH -- directly observed pattern in config.sh code.

## Minor Pitfalls

---

### Pitfall 11: Open Notebook Uses Streamlit -- WebSocket Proxy Required

**What goes wrong:** Open Notebook's UI is built with Streamlit (port 8502), which requires WebSocket support. If nginx doesn't include `Upgrade` and `Connection` headers for the Open Notebook location block, the UI loads the initial page but all interactive features (chat, editing, real-time updates) break silently.

**Prevention:** Add WebSocket headers to nginx location block for Open Notebook, same pattern as existing Open WebUI config: `proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade";`

**Confidence:** HIGH -- Streamlit always requires WebSocket proxying, well-documented.

---

### Pitfall 12: DB-GPT SiliconFlow API Key -- Unnecessary for Ollama-Only Setup

**What goes wrong:** DB-GPT's default Docker Compose requires `SILICONFLOW_API_KEY` env var. If the installer passes this, the user thinks they need a SiliconFlow account. If it's missing, the default entrypoint command may fail.

**Prevention:**
- Override the Docker `command:` in compose to use Ollama proxy mode instead of SiliconFlow
- Use SQLite mode with custom TOML config that only defines Ollama as provider
- Do NOT include `SILICONFLOW_API_KEY` in the AGMind env template
- Phase: **Config Generation** (custom command override in compose service definition)

**Confidence:** MEDIUM -- default compose requires it; custom config likely bypasses, needs testing.

---

### Pitfall 13: Health Check Logic -- New Services Not in phase_7_health

**What goes wrong:** `phase_7_health()` hardcodes `healthcheck_services="api db_postgres redis"` and conditionally adds `weaviate`. If Open Notebook, SurrealDB, and DB-GPT are added to compose but not to the health check logic, they can fail silently.

**Prevention:**
- Extend `phase_7_health()` to check COMPOSE_PROFILES for `open-notebook` and `dbgpt`
- Add Docker healthcheck definitions to compose for new services:
  - Open Notebook: `curl -f http://localhost:8502/_stcore/health`
  - DB-GPT: `curl -f http://localhost:5670`
  - SurrealDB: `curl -f http://localhost:8000/health`
- Phase: **Health Check** (extend health.sh conditionals)

**Confidence:** HIGH -- direct code review of health.sh confirms hardcoded service list.

---

### Pitfall 14: Memory Pressure -- Optional Tools Add 2-4 GB Combined

**What goes wrong:** AGMind already runs Dify (4 containers), PostgreSQL, Redis, vector DB, nginx, Open WebUI, plus native Ollama with a large model loaded in RAM. Adding Open Notebook (Streamlit app + SurrealDB) and DB-GPT (Python app + optional MySQL) adds 2-4 GB RAM consumption.

**Prevention:**
- Wizard should warn: "Open Notebook adds ~1 GB RAM. DB-GPT adds ~1-2 GB RAM."
- On 16 GB systems: recommend picking ONE optional tool, not both
- On 8 GB systems: do not offer optional tools at all (RAM already tight with base stack + small model)
- Add RAM estimate to wizard output before confirmation prompt
- Phase: **Wizard** (RAM-gated offering), **Preflight** (check available RAM minus base stack estimate)

**Confidence:** MEDIUM -- RAM estimates based on typical Python + database container footprints; exact numbers need measurement.

---

### Pitfall 15: CLI agmind.sh Unaware of Optional Tools

**What goes wrong:** `agmind status`, `agmind stop`, `agmind start` currently enumerate a fixed set of services. Optional tools won't appear in status output, won't be stopped/started by the CLI, and won't be included in backup operations.

**Prevention:**
- CLI must read COMPOSE_PROFILES from .env to discover active optional services
- `agmind status` should list Open Notebook and DB-GPT URLs when their profiles are active
- `agmind backup` must include SurrealDB data volume when open-notebook profile is active
- Phase: **CLI** (extend agmind.sh service awareness)

**Confidence:** HIGH -- direct implication of the profile-based architecture.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Preflight | DB-GPT image architecture mismatch on Apple Silicon | Check `docker manifest inspect` for arm64; warn user |
| Preflight | Port conflicts with new services (5670, 8000, 8502) | Add new ports to the port scanner |
| Preflight | Docker Compose version < 2.20.2 breaks `required: false` | Check `docker compose version` |
| Wizard | RAM insufficient for optional tools on 8-16 GB systems | Gate offering behind RAM threshold (>=32 GB for both) |
| Config Generation | Profile string assembly bugs (commas, name mismatch) | BATS tests for all profile combinations |
| Config Generation | Nginx fails with missing optional upstreams | Conditional template rendering based on active profiles |
| Config Generation | DB-GPT TOML config not generated | New template file + volume mount in compose |
| Config Generation | Open Notebook encryption key not persisted | Add to credentials.txt whitelist in config.sh |
| Config Generation | DB-GPT SiliconFlow key required by default entrypoint | Override compose command for Ollama-only mode |
| Compose Start | depends_on with profiled services breaks startup | Use `required: false` or remove cross-profile depends_on |
| Health Check | New services not checked by phase_7_health | Extend health.sh service list based on active profiles |
| Health Check | SurrealDB healthcheck endpoint undocumented | Test `/health` endpoint; fallback to running-state check |
| Backup | SurrealDB data not included in backup | Add surreal_data volume to backup script |
| CLI | `agmind status` doesn't show optional tools | Read COMPOSE_PROFILES to discover active services |
